# Accessibility audit: ADA / WCAG 2.2 AA compliance review

Audit the site against WCAG 2.2 AA (the current standard, published October 2023). ADA Title III compliance for websites is assessed against WCAG 2.1 AA at minimum; WCAG 2.2 AA is a superset and satisfies both. The previous accessibility work (Uto-0m32) targeted WCAG 2.1 AA. This audit should re-verify that work and extend it to cover WCAG 2.2 additions and any gaps found below.

## What is already in place (confirmed good)
- lang="en" on <html>
- Descriptive alt text on logo image
- <label for=...> properly associated with both textareas
- Semantic landmarks: <header>, <main>, <section>, <footer>
- Heading hierarchy: h1 in header, h2 in each section
- aria-live="polite" on the auto-detect badge
- aria-expanded / aria-controls on the Advanced Settings toggle
- aria-hidden="true" on decorative arrow icon
- aria-label on dynamically generated sort/visibility buttons (ui.js:722, 736, 802, 1521, 1538)
- focus-visible outline styles defined in CSS
- <kbd> elements and <details>/<summary> for keyboard shortcuts
- GitHub external link has rel="noopener"

## Specific findings to investigate and fix

**1. textarea:focus { outline: none } removes focus ring (WCAG 2.4.7, 2.4.11)**
css/main.css has textarea:focus-visible with a proper outline, but also textarea:focus { outline: none } (line 442). In browsers that do not support :focus-visible, this removes the focus indicator entirely for keyboard users tabbing to the textarea. Fix: remove the textarea:focus { outline: none } rule; the :focus-visible rule is sufficient.

**2. #copy-feedback and #discord-copy-feedback lack aria-live (WCAG 4.1.3)**
These divs are shown/hidden by toggling a CSS class. Without aria-live or role="status", screen readers will not announce copy success or failure. Fix: add role="status" aria-live="polite" aria-atomic="true" to both feedback divs in index.html.

**3. showMessage writes status into textarea value — not announced by screen readers (WCAG 4.1.3)**
Status messages (parse success, error, type detection) are set as the textarea's .value temporarily. A textarea is not a live region; screen readers will not announce the change. Fix: add a visually-hidden aria-live="assertive" region to index.html and route all showMessage calls to update it instead of (or in addition to) the textarea value.

**4. Color used alone to convey parse status (WCAG 1.4.1)**
showMessage sets textarea text color to red (error) or green (success) with no other visual distinction. Color alone must not be the sole means of conveying information. Fix: prefix messages with a text indicator (e.g. "Error: " / "Success: ") or an icon with aria-label, so status is conveyed independent of color.

**5. Color contrast — several values likely below 4.5:1 threshold (WCAG 1.4.3)**
Values to measure against their backgrounds:
  - Body text #d4d8de on #171719
  - Section h2 #8aafc8 on rgba(18,18,20)
  - Label/description text #7a8290 on section background
  - Footer text #7a8290 on body background
  - Advanced Settings small text #4e5560 (font-size ~0.68rem) — very likely fails, and small text has a 4.5:1 requirement
  - Placeholder text in textareas (browser default, typically lower contrast)
  - Feedback element text: #742a2a on #fed7d7, #744210 on #feebc8, #22543d on #c6f6d5 — these are light-on-light combinations; check contrast AND whether they meet requirements on the dark site background
  Use a tool such as the WebAIM Contrast Checker or browser DevTools to measure each pair.

**6. No skip navigation link (WCAG 2.4.1)**
Keyboard users must tab through the entire header before reaching the main content. A visually-hidden "Skip to main content" link as the first focusable element in <body> is the standard fix.

**7. Focus management after parse (WCAG 2.4.3)**
After clicking Parse Text, output appears in the output textarea but focus remains on the Parse button. Keyboard users must then tab to the output area to read or copy it. Consider moving focus to the output textarea, or ensuring the aria-live region (see finding #3) announces the result.

**8. WCAG 2.2 — Target Size Minimum (2.5.8, AA)**
New in WCAG 2.2: interactive targets must be at least 24×24 CSS pixels, or have sufficient spacing. Verify button heights and the Advanced Settings checkboxes and up/down sort arrows meet this on mobile viewports.

**9. WCAG 2.2 — Focus Not Obscured (2.4.11, AA)**
New in WCAG 2.2: when a UI component receives focus, it must not be fully hidden by author-created content. Verify no sticky elements (e.g. header) overlap focused controls when scrolling.

**10. prefers-reduced-motion not respected**
Check whether any CSS transitions or animations (hover effects, feedback fade-in/out) are gated with @media (prefers-reduced-motion: reduce). Users with vestibular disorders may have this set.

**11. External link opens in new tab without warning (WCAG 3.2.2)**
The GitHub issues link uses target="_blank" without notifying the user. Add visible text or a visually-hidden span indicating "(opens in new tab)".

**12. Dynamic Advanced Settings content — full keyboard and AT verification**
The section-order drag-equivalent (up/down buttons), visibility checkboxes, and unique-window input are all rendered via innerHTML in ui.js. Verify with a screen reader (VoiceOver on macOS/iOS, NVDA on Windows) that: all controls are reachable by keyboard, all have accessible names, and state changes (checkbox toggled, order changed) are announced.

**13. Windows High Contrast Mode / forced-colors**
The site is dark-mode only with no light-mode or forced-colors CSS. Windows High Contrast mode may override colors in unexpected ways. Test with the Windows High Contrast Black and White themes and fix any controls that become invisible.

## Recommended testing tools
- axe DevTools browser extension (automated — catches ~30-40% of WCAG issues)
- WAVE (WebAIM) browser extension
- Lighthouse accessibility audit in Chrome DevTools
- Manual keyboard-only navigation (Tab, Shift+Tab, Enter, Space, Esc, arrow keys)
- VoiceOver + Safari on iOS (since most mobile users are iOS)
- VoiceOver + Safari on macOS
- NVDA + Firefox on Windows (for AT diversity)
- WebAIM Contrast Checker for each color pair listed in finding #5

## Acceptance Criteria

1. All WCAG 2.2 AA success criteria pass automated axe/WAVE scan with zero violations. 2. All findings above are resolved or documented as accepted risk with rationale. 3. Keyboard-only navigation reaches every interactive element without a focus trap. 4. VoiceOver on iOS announces: parse result, copy result, auto-detected mode, and all Advanced Settings controls. 5. All text color pairs meet 4.5:1 contrast ratio (3:1 for large text ≥18pt or ≥14pt bold). 6. Skip navigation link present and functional.
